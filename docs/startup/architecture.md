# SPL-Flow Architecture Design

*February 2026 | SPL-Flow v0.1 MVP*

## 1. PocketFlow Usage Analysis

### Should the SPL engine core use PocketFlow?

**Short answer: No for the core engine, Yes for the platform layer.**

The SPL engine's internal pipeline (`parse → analyze → optimize → execute`) is a **fixed, deterministic sequence** with no branching, retries, or human checkpoints. PocketFlow would add framework overhead with no benefit. The CTE parallel execution is already handled natively with `asyncio.gather()`.

| SPL Engine Component | PocketFlow? | Reason |
|---------------------|-------------|--------|
| `Lexer → Parser → Analyzer → Optimizer` | No | Fixed pipeline, zero branching |
| `Executor._execute_ctes_parallel` | No | `asyncio.gather` already handles this |
| `Executor.execute` single PROMPT | No | Deterministic, one code path |
| `cli._cmd_execute` loop | No | Simple iteration over plans |

**Where PocketFlow adds real value — the SPL-Flow orchestration layer:**

| SPL-Flow Component | PocketFlow? | Why Yes |
|-------------------|-------------|---------|
| `text → SPL` translation | **Yes** | Needs retry on parse failure, error context |
| `validate → retry → validate` loop | **Yes** | Natural conditional branching |
| `execute → sync OR async delivery` | **Yes** | Two divergent paths |
| Job queue worker | **Yes** | Background async processing |
| Scheduled execution (v0.4) | **Yes** | Cron-like trigger → execute → deliver |

## 2. SPL-Flow Architecture

### Pipeline Overview

```
User (Streamlit)
      │
      │ free-form text + optional context doc
      ▼
┌─────────────────────────────────────────────────────────────────┐
│                    PocketFlow Orchestration                      │
│                                                                  │
│  Text2SPLNode ──► ValidateSPLNode ──► ExecuteSPLNode           │
│       ▲                │                    │                    │
│       └── retry ───────┘ (up to 3x)         │                    │
│                         └─ "execute"        ▼                    │
│                                   ┌─────────────────┐           │
│                                   │  delivery_mode  │           │
│                                   └─────────────────┘           │
│                                      │           │               │
│                              "sync"  │           │ "async"       │
│                                      ▼           ▼               │
│                              SyncDeliver   AsyncDeliver          │
│                              (show in app) (save + email)        │
└─────────────────────────────────────────────────────────────────┘
      │
      ▼
SPL Engine (spl-llm package)
  Lexer → Parser → Analyzer → Optimizer → Executor
                                              │
                    ┌─────────────────────────┤
                    │ CTE sub-plans (parallel) │
                    │  asyncio.gather()        │
                    └─────────────────────────┘
                              │
              ┌───────────────┴────────────────┐
              ▼               ▼                ▼
       Ollama adapter   OpenRouter adapter  Claude CLI
       (local models)   (100+ models)       (subscription)
```

### Shared Store Schema

The PocketFlow shared store passed through all nodes:

```python
shared = {
    # Input
    "user_input": str,           # Free-form natural language query
    "context_text": str,         # Optional pasted document
    "adapter": str,              # "claude_cli" | "openrouter" | "ollama"
    "delivery_mode": str,        # "sync" | "async"
    "notify_email": str,         # For async delivery
    "spl_params": dict,          # Extra context params (key: value)
    "cache_enabled": bool,       # Default False for development

    # Generated by Text2SPLNode
    "spl_query": str,            # Generated SPL code
    "retry_count": int,          # How many generation attempts
    "last_parse_error": str,     # Error message for retry context

    # Generated by ValidateSPLNode
    "spl_ast": Program,          # Parsed AST
    "spl_warnings": list[str],   # Non-fatal warnings

    # Generated by ExecuteSPLNode
    "execution_results": list[dict],  # Per-PROMPT results
    "primary_result": str,            # Final PROMPT output (shown to user)

    # Delivery
    "delivered": bool,
    "output_file": str,          # Path to saved file (async mode)
    "email_sent": bool,

    # Errors
    "error": str,                # Fatal error if any
}
```

## 3. Text2SPL Design

The most critical component — converting natural language to valid SPL.

### Approach: Few-Shot LLM Translation with Retry

```
User: "List 10 Japanese water radical kanji with meanings and German translations"
         │
         ▼
Text2SPLNode (max_retries=3)
         │
         │ LLM call with system prompt + few-shot examples
         ▼
Generated SPL:
  PROMPT water_kanji_table
  WITH BUDGET 8000 tokens
  USING MODEL "anthropic/claude-sonnet-4-5"
  WITH characters AS (
      PROMPT cjk_analysis
      USING MODEL "qwen2.5"
      ...
  ),
  german AS (
      PROMPT german_trans
      USING MODEL "mistral"
      ...
  )
  SELECT ...
  GENERATE compose_table(..., "instruction")
  WITH OUTPUT BUDGET 2000 tokens, FORMAT markdown;
         │
         ▼
ValidateSPLNode: parse() succeeds → proceed
                 parse() fails  → retry with error message
```

### Model Routing Logic (codified in Text2SPL prompt)

| Task Type | Recommended Model | Reason |
|-----------|------------------|--------|
| CJK (Chinese/Japanese/Korean) | `qwen2.5` | Trained on large CJK corpus |
| European languages (German, French, etc.) | `mistral` | Strong EU language training |
| Code review / generation | `anthropic/claude-sonnet-4-5` | Best code reasoning |
| Encyclopedic / cultural facts | `meta-llama/llama-3.1-8b-instruct` | Broad knowledge |
| Final synthesis / composition | `anthropic/claude-sonnet-4-5` | Superior structure |
| Simple factual / fast tasks | `meta-llama/llama-3.1-8b-instruct` | Cost-efficient |
| Single-model simple query | `anthropic/claude-sonnet-4-5` | Default safe choice |

**Decision rule for CTE vs single PROMPT:**
- Single PROMPT: task is monolingual, single domain, one model handles it well
- Multi-CTE: task requires 2+ specialist models (e.g., CJK + European language + synthesis)

## 5. Async Mode Design

For v0.1 MVP, async mode is simplified:
- User clicks Execute → background thread runs SPL pipeline
- Result saved to `/tmp/spl_flow_result_<timestamp>.md`
- Download button appears when complete
- Email notification: placeholder (SMTP integration in v0.2)

For v0.4 Platform Alpha:
- SQLite job queue (`spl-flow submit`)
- Background worker process
- Email delivery via SMTP
- Scheduled execution (`--schedule "every Monday 8am"`)

### Innovation 1 — MoM: Mixture-of-Models (Conceptual Paradigm)

**The insight:**
SQL (1970) made data retrieval declarative. SPL makes LLM orchestration declarative.
The natural inference paradigm that emerges is **Mixture-of-Models (MoM)** — heterogeneous
specialist LLMs composed at the task level via explicit declarative routing.

This is analogous to — but meaningfully superior to — the established
Mixture-of-Experts (MoE) architecture used inside models like Mixtral and GPT-4.

**MoE vs MoM comparison:**

| Dimension | MoE (e.g. Mixtral 8×7B) | MoM (SPL engine) |
|-----------|--------------------------|------------------|
| Expert granularity | Sub-network within one model | Complete independent LLM |
| Routing mechanism | Learned gating function (trained weights) | Declarative SPL script (explicit, auditable) |
| Routing update | Requires full retraining | Edit the SPL script or YAML registry |
| Transparency | Black box | Human-readable, inspectable |
| Add new expert | Retrain entire model | Add `USING MODEL "..."` to SPL |
| Mix proprietary + open | Impossible | Native — any adapter |
| Activation granularity | Per-token | Per-task / per-CTE |
| Cost visibility | Opaque | Per-model token counts + cost logged |
| Domain specialization | Constrained by joint training | True specialists (qwen→CJK, mistral→EU, deepseek→code) |

**Why MoM is "poor man's MoE but actually better":**

MoE requires all experts to share architecture, training regime, and deployment hardware.
MoM assembles pre-trained specialists dynamically at inference time. A linguist, a coder,
and a reasoner — each the best in its class — composing a single result. No joint training
needed. No architecture constraints. Fully configurable by the user.

**MoM as a super-model:**
A well-authored SPL script is effectively a configurable, dynamic super-model whose
behavior can be updated by editing a text file — no GPU, no retraining, no deployment.

**The Text2SPL layer closes the loop:**
When Text2SPL generates the SPL routing automatically from a natural language query,
the Text2SPL LLM becomes the *learned router on top of MoM* — a meta-MoE at the
system level, with full transparency and human override capability.

```
User Query (NL)
     │
     ▼
Text2SPL (learned router — decides which MoM configuration to generate)
     │
     ▼
SPL Script (explicit MoM routing — declarative, auditable, editable)
     │
     ▼
┌────────────────────────────────────────┐
│  MoM Execution (parallel CTE dispatch) │
│  ┌──────────┐ ┌─────────┐ ┌─────────┐ │
│  │ qwen2.5  │ │ mistral │ │ llama3.1│ │  ← specialists
│  │ (CJK)   │ │ (EU)    │ │ (facts) │ │
│  └──────────┘ └─────────┘ └─────────┘ │
└────────────────────────────────────────┘
     │
     ▼
claude-sonnet (synthesis — composes final result)
     │
     ▼
Composed output (table / report / analysis)
```

**Proven on Feb 14 2026:**
Chinese radical 日-family table — 3 specialist models dispatched in parallel,
synthesized by Claude into a 6-column multilingual scholarly table.
Validated on Ollama (local/free), OpenRouter (cloud/$0.002), and Claude CLI.

### Innovation 2 — `USING MODEL auto` (Implementation Excellence)

**The vision:**
Users should not need to know which model is best for their task. SPL should route
intelligently based on codified knowledge of the LLM landscape.

```sql
-- Current (explicit routing):
USING MODEL "qwen/qwen-2.5-72b-instruct"

-- Future (intelligent auto-routing):
USING MODEL auto               -- best model for detected task type
USING MODEL auto(cost=low)     -- optimize for price (use efficient models)
USING MODEL auto(speed=high)   -- optimize for latency (fastest available)
USING MODEL auto(privacy=local) -- Ollama only, zero data egress
USING MODEL auto(quality=max)  -- best quality regardless of cost
```

**Implementation design:**

```
spl/model_router.py
  └── TaskTypeDetector
        ├── CJK detector (Unicode range U+4E00–U+9FFF in prompt)
        ├── Code detector (keywords: def, class, function, import, bug, review)
        ├── EU language detector (German/French/Spanish keywords)
        ├── Math detector (equations, symbols, proof keywords)
        └── Default: general reasoning

  └── ModelRegistry (spl/config/model_registry.yaml)
        ├── Source: HuggingFace Open LLM Leaderboard (updated quarterly)
        ├── Source: LMSYS Chatbot Arena Elo scores
        ├── Per task type: ranked list of models by adapter
        └── Constraints: cost_per_token, avg_latency_ms, context_window

  └── resolve_auto_model(task_type, adapter, constraints) → model_id
        └── Intercepts model == "auto" in Optimizer before execution
```

**Registry format (`spl/config/model_registry.yaml`):**

```yaml
# Updated from HuggingFace Leaderboard + LMSYS Arena Elo
# Last updated: 2026-02-14
routing:
  cjk:
    openrouter: ["qwen/qwen-2.5-72b-instruct", "qwen/qwen-2.5-7b-instruct"]
    ollama:     ["qwen2.5", "qwen2.5:3b"]
    default:    "anthropic/claude-sonnet-4-5"
  code:
    openrouter: ["deepseek/deepseek-coder-v2", "qwen/qwen-2.5-coder-32b"]
    ollama:     ["qwen2.5-coder", "deepseek-coder-v2"]
    default:    "anthropic/claude-sonnet-4-5"
  eu_language:
    openrouter: ["mistralai/mistral-large", "mistralai/mistral-7b-instruct"]
    ollama:     ["mistral", "mistral-nemo"]
    default:    "anthropic/claude-sonnet-4-5"
  reasoning:
    openrouter: ["anthropic/claude-sonnet-4-5", "deepseek/deepseek-r1"]
    ollama:     ["deepseek-r1", "qwen3"]
    default:    "anthropic/claude-sonnet-4-5"
  general:
    openrouter: ["anthropic/claude-sonnet-4-5", "google/gemini-2.0-flash"]
    ollama:     ["qwen2.5", "llama3.1"]
    default:    "anthropic/claude-sonnet-4-5"
```

**Why this matters:**
The registry is a YAML file — updated without code changes. As the LLM landscape evolves
(new models, new Elo rankings, new price points), SPL users automatically benefit.
`USING MODEL auto` makes SPL future-proof: the routing intelligence upgrades independently
of user scripts.

### Innovation 3 — Text2SPL Fine-Tuned LFM (Liquid AI Partnership)

**The vision:**
A tiny, fast, specialized model (1-3B parameters) trained exclusively to translate
natural language queries into valid SPL scripts. Runs locally, in-browser, on-device.
Zero cloud dependency for the translation step.

**Why Liquid AI's LFM family is the ideal foundation:**

| Property | Why it matters for Text2SPL |
|----------|----------------------------|
| LFM 1.3B compact size | Runs on CPU, in browser (WASM), on mobile |
| Liquid Neural Network architecture | Efficient for structured sequence translation |
| Fast inference | ~80ms NL→SPL translation vs 2-3s for GPT-4 |
| On-device / privacy-first | No query data leaves the user's machine |
| Fine-tunable | Liquid AI supports fine-tuning on domain-specific tasks |

**This is a killer use case for Liquid AI:**
- Demonstrates LFM utility beyond chat — specialized translation task
- On-device NL→structured-query is a massive market (NL→SQL has $1B+ tooling market)
- NL→SPL is a new category: NL→LLM-orchestration-script
- Ramin Hasani connection: reach out once SPL-Flow MVP is solid

**Training data strategy:**

```
Phase 1 — Synthetic Bootstrap (automated)
  ├── Generate 10,000+ (NL query, SPL script) pairs via GPT-4/Claude
  ├── Validate every SPL with `spl validate` — discard invalid
  ├── Diversity: single-model, multi-CTE, various domains, adapters
  └── Clean labeled dataset ready for fine-tuning

Phase 2 — Fine-tune LFM 1.3B
  ├── Task: sequence-to-sequence translation (NL → SPL)
  ├── Grammar-constrained decoding (beam search stays within SPL grammar)
  ├── Evaluation metric: parse success rate + semantic intent alignment
  └── Target: >95% valid SPL on held-out test set

Phase 3 — RLHF / DPO Alignment
  ├── Human preference: (query, SPL_A, SPL_B) → preferred
  ├── Aligns generated SPL with user intent beyond syntactic correctness
  └── Closes gap between "valid SPL" and "correct SPL for this task"

Phase 4 — Integration
  ├── Packaged as spl-flow-translator (optional pip install)
  ├── Falls back to GPT-4/Claude if LFM not available
  └── Runs fully locally when LFM is present
```

**The full Text2SPL LFM stack:**

```
User types: "List 10 Chinese characters with water radical,
             meanings, German translations, cultural insights"
         │
         ▼ (80ms, local, private)
Text2SPL LFM 1.3B
  └── Grammar-constrained beam search
         │
         ▼
Valid SPL (USING MODEL auto → routes to qwen2.5, mistral, llama3.1)
         │
         ▼
SPL Engine: parallel CTE dispatch
         │
         ▼ (future multi-modal CTEs)
┌─────────────────────────────────────────────────────┐
│  PROMPT table   USING MODEL "claude-sonnet"          │
│  WITH chars AS (PROMPT ... USING MODEL "qwen2.5")    │
│  WITH audio AS (PROMPT ... USING MODEL "elevenlabs") │
│  WITH image AS (PROMPT ... USING MODEL "sdxl")       │
│  WITH strokes AS (PROMPT ... USING MODEL "cjk-anim") │
└─────────────────────────────────────────────────────┘
         │
         ▼
Composed result: table + audio pronunciation + visual mnemonic
```

**Why this makes ZiNets Chinese Learning Platform unique and unreplicable:**
- Every competitor uses one model for everything
- SPL-Flow routes each sub-task to the world's best specialist model
- The orchestration is invisible to the learner — they see one beautiful result
- As better models appear, the YAML registry updates — the platform gets smarter automatically
- Text2SPL LFM makes this accessible to non-engineers via plain English
